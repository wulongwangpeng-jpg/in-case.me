## 一、产品概述

### 1.1 我们解决什么问题

中国有 **10.8 亿网民**，平均每人拥有 **20+ 个数字账户**——微信、支付宝、网银、股票、基金、游戏、社交、云盘、订阅服务……

**人没了，账号还在。** 家属不知道有哪些账号、不知道密码、不知道找谁处理。每一个账号背后都是真金白银或重要回忆，但绝大多数人从没整理过。

现有的"解决方案"全都不 work：

| 方案 | 问题 |
|------|------|
| 写在纸上告诉家人 | 纸会丢、会忘、账户一直在变 |
| 用密码管理器 | 家属不知道怎么用、不知道主密码 |
| 找律师立遗嘱 | 贵（几千起）、重财产不重数字资产、年轻人不会去 |
| 平台自身的"遗产联系人" | 只有苹果/Google 有，国内平台几乎不支持 |

### 1.2 产品三步骤

```
① 资产盘点              ② 心底话              ③ 设置信使
盘一遍你有哪些数字资产   给重要的人写封信       指定信任的人，给他们"钥匙"
AI 聊天式引导 + 直接录入  也可写下处置说明       你安好时他们打不开
                                                你失联 180 天后才能提取
```

### 1.3 为什么是现在

- 中国第一代网民（70后/80后）正在进入中年，开始认真思考"万一"
- 年轻人（90后/00后）数字资产复杂度远超上一代：自媒体账号、游戏皮肤、数字藏品、知识付费内容
- "数字遗产"概念正在进入公众视野（民法典已明确虚拟财产可继承）
- 但市面上**没有一个普通人用得起的零知识加密工具**

## 二、品牌叙事与定位

### 2.1 核心定位

**"温暖地交代一切"** — 不是冷冰冰的遗嘱工具，而是有温度的数字交代。

品牌叙事保持这个定位不变（战略决策 #5）：首页加工具属性入口降低进入门槛，但核心叙事不降级为"自动续费退订指南"。

### 2.2 品牌调性

| 维度 | 做法 |
|------|------|
| **情感基调** | 温暖、仪式感、不制造焦虑 |
| **文案风格** | 中性化，避免"死亡""遗产""失联"等触发词 |
| **视觉语言** | 柔和配色、圆润卡片、渐进式引导 |
| **信任建设** | 开源 + 零知识加密演示 + 安全白皮书式解释 |

### 2.3 双市场定位

| | 海外站 (wanyine.com) | 国内站 (wanyine.cn) |
|---|---|---|
| **模式** | 纯付费订阅 | 完全免费 |
| **叙事** | "Digital Estate Planning, Simplified" | "温暖地交代一切" |
| **卖点** | Privacy-first, Zero-knowledge | AI 辅助 + 情感温度 |
| **目标用户** | Privacy-conscious 英语用户 | 25-45 岁城市互联网用户 |
| **策略** | 海外赚美金，国内赚口碑 | 共享同一套代码 |

## 三、完整功能矩阵

### 3.1 已上线功能（v1.2）

#### 🏠 首页
- **Hero 区**：品牌价值主张 + CTA 按钮
- **信任幻灯片**（TrustSlideshow）：5 步侧边滑块解释安全模型
- **加密演示**（EncryptionDemo）：实时 AES-256-GCM 加密/解密动画——输入任何文字即时看到密文
- **续期签到卡片**（RenewalCheckin）：显示剩余安全天数、两档续期（90 天「我很好」/ 45 天「有点忙」）、阈值调整
- **引导弹窗**（OnboardingGuide）：首次访问 4 张全屏引导卡片
- **三大功能卡片**：资产盘点 / 心底话 / 设置信使

#### 📋 资产盘点 (`/inventory`)
- **直接录入模式**：分类标签（金融资产 / 投资理财 / 创作内容 / 游戏资产）+ 平台名 + 账号 + 处理方式 + 备注
- **AI 辅助模式**：聊天式引导对话，AI 逐个领域帮你梳理遗漏的资产
  - 启动对话 → 连续问答 → 自动总结提取结构化资产列表
  - 支持 Demo 模式（无 API Key 时使用内置模拟回复）
- **加密保存**：密码设置/密码生成器 → PBKDF2 + AES-256-GCM 浏览器端加密 → 上传密文
- **保险库列表**（VaultList）：按类型过滤、输入密码解密查看、删除

#### ✉️ 心底话 (`/letters`)
- **直接书写模式**：写给重要的人的信
- **AI 辅助写信模式**：对话式引导 → 生成草稿 → 支持重写
  - Demo 模式同样可用
- **加密保存/解密查看**：与资产盘点共享加密管线

#### 🔐 信使系统 (`/settings` → CredentialManager)
- **创建信使凭证**：
  - 填写信使手机号/邮箱 + 关系标签
  - 选择可查看的密文范围（细粒度权限，每位信使可见不同内容）
  - 系统生成双码：人类可读短码（如 `枫叶-长颈鹿-37`）+ SHA-256 安全令牌
  - 可选短信/邮件通知信使
- **信使列表管理**：查看/吊销/恢复凭证
- **QR 码分享**（QRShare）：生成包含链接+短码的二维码，可截图分享
- **去晦气化文案**：邀请通知用"数字备忘录"替代"数字遗产"

#### 🔓 信使提取 (`/unlock?token=`)
- 信使通过链接+短码访问
- 输入密码解密查看被分配的内容
- **AI 合规提取指南**：根据资产平台名自动生成合规的继承/提取流程
- 支持导出解密后的内容
- 纯时间戳访问控制：服务端仅判断 `lastActiveTime + safeThresholdDays < NOW()`

#### 🤝 信使接受 (`/accept?token=`)
- 信使打开邀请链接后的确认页
- 安抚文案 + 仪式感设计
- 确认接受后即成为正式信使

#### ⚙️ 安全设置 (`/settings`)
- **订阅面板**（SubscriptionPanel）：年付 $23 / 终身 $198，PayPal 支付
- **信使管理**（CredentialManager）：创建/列出/吊销/恢复凭证
- **灾备面板**（BackupPanel）：一键导出全部加密数据为 JSON，支持离线恢复导入

#### 👤 身份与绑定
- **极简 Cookie UUID 模式**：无需注册，首次访问自动分配 `wanyi_user_token`
- **可选绑定手机号/邮箱**：用于跨设备恢复和信使通知
- **BindModal**：验证码发送（60s 频率限制）→ 验证 → 绑定
- **解绑**：`/api/auth/unbind`
- **导航栏状态**：
  - 未绑定 → 引导绑定（不会误走 login 死胡同）
  - 已绑定 → 显示脱敏联系方式 + 下拉菜单（更换绑定/解除绑定）

#### 🤖 AI 能力
- **AI 资产盘点对话**：`POST /api/ai/inventory` — 多轮对话，自动结构化提取
- **AI 写信辅助**：`POST /api/ai/letter` — 对话引导 + 草稿生成 + 重写
- **AI 合规指南**：`POST /api/ai/compliance-guide` — 按资产平台名生成继承流程，内置中英文常见平台 Demo 库
- **统一客户端**：`src/lib/ai-client.ts` — OpenAI 兼容接口，换环境变量即可切模型
- **当前使用**：DeepSeek V4 Pro（`deepseek-chat`）

#### 📡 通知系统
- **Twilio 短信**：信使邀请通知 + 死信核验全链路
- **Resend 邮件**：HTML 邮件模板（品牌头 + 绿底验证码 + 干净排版）
- **Provider 抽象层**：`src/lib/notify.ts` — 按环境变量自动切换 Twilio/Resend/Demo
- **Demo 降级**：未配置时自动 console.log，不阻塞功能

#### 💰 支付系统
- **PayPal Orders API v2**：终身计划一次性付款（$198）
- **PayPal Subscriptions API v1**：年费循环扣款（$23/年）
- **Webhook 处理**：`POST /api/paypal/webhook` — 订阅激活/扣款失败/取消/过期
- **乐观更新 UI**：续期按钮点击即变，卡片不消失

#### ⏰ 死信处理全链路（Cron Job）
- **触发**：Vercel Cron 每天 9:00 AM 执行
- **流程**：
  1. 扫描超阈值用户（默认 180 天未活跃）
  2. 海外站过滤未订阅用户
  3. 对每个信使，分阶段通知：
     - **0h**：首次短信/邮件 → "您的朋友委托您保管一份数字备忘录，现已到达约定的查看时间"
     - **24h**：第一次提醒
     - **48h**：最后一次提醒
     - **72h+ 无回复**：自动交付提取链接
  4. 信使回复 1 → 自动续期 | 回复 2 → 下发提取链接
- **文案极度中性化**：避免"失联""死亡""提取资产"等触发词

#### 📄 法律页面
- 隐私政策 (`/privacy`)
- 服务条款 (`/terms`)
- FAQ (`/faq`)
- Cookie 同意横幅

#### 🌐 国际化
- **中英双语完整覆盖**：`src/i18n/zh.ts`（~875 行）+ `src/i18n/en.ts`（~820 行）
- **自动检测**：`?lang=` query → cookie → navigator.language
- **React Context Provider**：`useI18n()` hook 全局可用

## 四、用户旅程地图

### 4.1 新用户首次体验

```
访问首页
  └─ 看到 Hero 区：「温暖地交代一切」
  └─ 加密演示动画吸引注意
  └─ 新手引导弹窗（4 张卡片）
  └─ 三大功能卡片引导下一步

进入资产盘点 → AI 聊天式引导 → 发现原来有这么多资产 → 设置密码 → 加密保存
进入心底话 → 给重要的人写一封信 → 加密保存
进入设置 → 创建信使 → 填写信使联系方式 → 选择可查看的内容 → 发送邀请通知

续期签到 → 看到剩余天数 → 点击「我很好」→ 卡片即时更新
```

### 4.2 信使旅程

```
收到短信/邮件 → 「您的朋友委托您保管一份数字备忘录」
  └─ 打开链接 → 安抚文案页：「别紧张，你的朋友一切安好」
  └─ 确认接受 → 成为正式信使
  └─ 日常：什么都不会发生

─── 用户失联超过 180 天 ───

收到死信通知（0h） → 可以回复 1（安好续期）/ 2（失联提取）
收到提醒（24h）→ 「您的朋友可能暂时无法联系，请确认他的状态」
收到最后提醒（48h）→ 「这是最后一次提醒，如您仍无法联系到他……」
72h 无回复 → 自动交付提取链接
  └─ 打开提取页 → 输入密码 → 解密查看 → AI 合规提取指南 → 导出
```

### 4.3 付费用户旅程

```
免费体验 → 创建 3 条以上心底话 → 需要更多信使
  └─ 进入设置 → 看到订阅方案
  └─ 年付 $23/年 或 终身 $198
  └─ 点击 → PayPal 弹窗 → 支付完成
  └─ Webhook 激活 → 订阅状态更新 → 解锁全部功能
```

### 5.3 数据库设计（6 张表）

```
users
  ├─ id (uuid, PK)
  ├─ email, phone (可选绑定)
  ├─ nickname
  ├─ lastActiveTime (timestamp, 续期时更新)
  ├─ safeThresholdDays (默认 180)
  └─ createdAt, updatedAt

vaults
  ├─ id (uuid, PK)
  ├─ userId (FK → users)
  ├─ vaultType (asset_inventory | farewell_letter | wishlist)
  ├─ aliasName, memoryHint, categoryTag
  ├─ encryptedContent (text, Base64 密文)
  ├─ encryptionSalt, encryptionIv (Base64)
  ├─ encryptionVersion (默认 1)
  └─ createdAt, updatedAt

credentials
  ├─ id (uuid, PK)
  ├─ userId (FK → users)
  ├─ messengerEmail, messengerPhone, messengerPhone2
  ├─ messengerRelation, messengerLabel
  ├─ humanCode (人类可读短码, 如 "枫叶-长颈鹿-37")
  ├─ accessToken (SHA-256 安全令牌)
  ├─ status (active | revoked | delivered)
  └─ accessCount, lastAccessedAt

credential_vaults (多对多)
  ├─ credentialId (FK → credentials)
  ├─ vaultId (FK → vaults)
  └─ createdAt

access_logs (审计日志)
  ├─ credentialId (FK)
  ├─ userId (FK)
  ├─ eventType (12 种事件类型)
  ├─ result (allowed | denied)
  ├─ reason, ipAddress, userAgent
  └─ createdAt

subscriptions
  ├─ userId (FK → users)
  ├─ paypalSubscriberId
  ├─ paypalSubscriptionId (unique)
  ├─ plan (annual | lifetime)
  ├─ status (active | canceled | expired)
  ├─ currentPeriodEnd
  └─ createdAt, updatedAt

verification_codes
  ├─ id, target, codeHash
  ├─ purpose (bind | login)
  ├─ expiresAt, used
  └─ createdAt
```

### 5.4 API 设计原则

- **RESTful**：资源命名清晰（`/credentials/[id]`），状态码规范
- **无状态**：Cookie UUID 标识用户，不需要 session
- **防滥用**：验证码 60s 频率限制 + 5 分钟过期
- **审计完备**：关键操作写入 `access_logs`
- **Cron 保护**：`CRON_SECRET` 校验
- **幂等**：PayPal capture-order 幂等写入，防止重复扣款

## 六、安全模型详解

### 6.1 零知识加密 — 核心原理

```
用户密码 ──→ PBKDF2 (100,000 次迭代, SHA-256) ──→ AES-256-GCM 密钥
                                                      │
明文内容 ─────────────────────────────────────────────┼──→ 密文（上传服务器）
                                                      │
                                         盐 + IV（上传服务器）
```

**关键约束：**
- 用户密码**永不上传**服务端
- 加密/解密 **100% 在浏览器端**完成（Web Crypto API）
- 服务端只存储：密文（ciphertext）+ 盐（salt）+ IV + 版本号
- **我们自己也打不开用户数据** — 这是合规优势，也是信任基础

### 6.2 信使双码机制

```
人类可读短码：「枫叶-长颈鹿-37」
  ├─ 组成：形容词(16) × 动物(16) × 数字(10-99) ≈ 25,000 组合
  ├─ 用途：信使口头告知/二维码分享
  └─ 安全性：需要配合链接中的 accessToken

安全令牌：SHA-256(userId + humanCode + timestamp + randomUUID) → Base64 → 64字符
  ├─ 用途：URL 参数，验证信使身份
  └─ 安全性：碰撞概率极低，泄露后可吊销重发
```

### 6.3 纯时间戳访问控制

服务端**不做权限判断**，只做一个计算：

```
NOW() > lastActiveTime + safeThresholdDays ?
  → YES → 信使可以提取
  → NO  → 拒绝（你安好时，任何人都打不开）
```

没有"管理员可以查看"的后门，没有"法院命令可以解密"的机制 — 因为我们技术上做不到。

### 6.4 多层防护

| 层 | 措施 |
|----|------|
| **传输层** | HTTPS (Vercel 自动) |
| **存储层** | PostgreSQL 只存密文 + Supabase 加密 at rest |
| **应用层** | AES-256-GCM 认证加密（防篡改） |
| **身份层** | Cookie UUID + 可选手机/邮箱绑定 |
| **访问层** | 纯时间戳控制，无后门 |
| **审计层** | access_logs 记录所有关键操作 |
| **导出层** | 灾备导出为加密 JSON，不解密密文 |

### 6.5 安全边界说明

| 我们能做什么 | 我们不能做什么 |
|-------------|---------------|
| 存储密文 | 读取明文 |
| 验证时间戳 | 重置密码 |
| 发送通知 | 恢复丢失的密码 |
| 吊销凭证 | 在用户活着时让信使提取 |
| 记录审计日志 | 绕过加密查看内容 |

## 十六、补充：你可能没想到的事

### 16.1 技术层面

**1. 当前首页是 `"use client"` — SEO 是盲区。**
Google 爬虫能渲染 JS，但排名权重低于 SSR。首页是最大的流量入口，应改为服务端渲染。改造成本 ~2h。

**2. 没有 staging 环境。**
目前 `git push → production` 是直通车。建议至少加 Vercel Preview Deployments（PR 触发，免费），在合并前验证。

**3. API 没有 rate limiting。**
验证码有 60s 限制，但其他端点（如 `/api/unlock`）没有。Vercel 免费层不提供 WAF，如果被恶意扫描可能产生费用或被滥用。建议至少对 `/api/unlock` 加 IP 级别限制。

**4. 数据库没有备份策略验证。**
Supabase 说它有自动备份，但你恢复过吗？建议手动做一次恢复演练。Free tier 的备份策略可能不如预期。

**5. AI API Key 写在环境变量里可以 SSR 调用 — 但要防止客户端泄露。**
当前所有 AI 调用都是 `POST /api/ai/*` → 服务端调用 DeepSeek，安全。但 `NEXT_PUBLIC_` 前缀的变量会暴露给客户端 — 确保 AI_API_KEY 永远不以 `NEXT_PUBLIC_` 开头。

**6. 加密版本号 forward compatibility。**
当前 `CURRENT_VERSION = 1`。如果将来升级加密算法（如 AES-256-GCM → 某种后量子算法），需要支持解密旧版本密文后用新算法重新加密。建议在保险库列表中标记哪些条目是旧版本加密的，引导用户重新加密。

### 16.2 产品层面

**7. 密码丢失 = 数据永久丢失。**
这是零知识的代价，但也是最大的用户投诉来源。你需要在每个设置密码的地方用醒目的警告告知用户。目前 UI 有提示，但不够强调。建议：密码设置页加红色警告框 + 强制用户输入「我理解密码丢失后数据无法恢复」才能继续。

**8. 信使死了怎么办。**
用户可以设置多个信使，但目前没有"信使也失联了"的兜底机制。建议：允许设置备用信使链，或允许多信使中任意 N 个联合解密（Shamir Secret Sharing）。

**9. 用户真的会来续期吗。**
90 天续期是一个产品假设。如果用户忘了呢？建议：在第 80/85/90 天给用户本人发提醒（邮件/短信），这比等 180 天后通知信使更符合用户期望。

**10. 英文版不是翻译，是重写。**
当前英文语言包是中文的翻译版。但品牌叙事、文化语境都需要独立设计。比如 "温暖地交代一切" 直译成 "Warmly Explain Everything" 会让人困惑。海外应该用 "Your digital life, taken care of" 或类似叙事。

### 16.3 运营层面

**11. Product Hunt 不只是发布，是一场战役。**
需要提前准备的：
- 一个 Hunter（有 PH 影响力的用户）帮你发布
- 首小时投票策略（PH 算法高度权重首小时表现）
- 英文截图（5 张）+ 产品视频（推荐）
- Maker 评论：讲你的故事，为什么做这个产品
- 回复每一条评论（这是获取早期用户最宝贵的时刻）
- 如果进入 Top 5 → 会有持续的 organic traffic

**12. 时区问题。**
你的 Cron 是每天北京时间 9:00 AM。如果用户在美国西海岸，那是前一天 17:00/18:00（取决于夏令时）。死信通知的"24h""48h""72h"是按绝对时间算的，不受影响，但你的客服和 Product Hunt 发布时间需要考虑时区——PH 按太平洋时间零点重置。

**13. 退款政策。**
目前没有退款流程。PayPal 允许 180 天内 dispute。建议在服务条款中写清楚退款政策，并准备一个简单的退款流程（PayPal Dashboard 手动操作即可，初期不需要自动化）。

**14. 用户数据删除请求。**
GDPR/CCPA 要求 30 天内响应用户数据删除请求。目前没有一键删除账号的 API。建议添加 `/api/me/delete` 端点，清除用户的所有 vault、credential、subscription、verification_codes 记录。

**15. 你的成本不只是服务器。**
一个月 ¥150 是纯基础设施。真正的成本是你自己的时间——客服、运维、迭代。如果你把时间算进去（按市场价 ¥200/h），每月投入 20h = ¥4000。只要有了 50 个付费用户，这个投入就开始有了回报。

### 16.4 增长层面

**16. 信使裂变的质量远高于广告，但量少。**
每个用户平均可能只有 1-2 个信使。信使 → 注册的转化率可能 10-20%。这是高质量但低流量的增长引擎。你需要额外一个高流量引擎来喂顶部漏斗——内容营销（SEO 博客）是最好的选择。

**17. "死了吗"有 50 万用户 — 不要低估中国市场。**
虽然它没有加密、没有 AI、没有信使体系，但它验证了需求存在。国内站（免费）上线后，可以从小红书/豆瓣的"数字遗产"话题切入，用「AI 帮你盘点数字资产」作为社交传播钩子。

**18. Open source as marketing。**
AGPL-3.0 意味着商业使用需要授权。如果你围绕"零知识加密数字遗产"建立技术博客和开源社区，会吸引技术影响力人群——他们是最可能成为早期付费用户的人（关心隐私，愿意为安全付费）。

### 16.5 法律层面

**19. 信使提取后，资产处置是信使的责任——但你需要在服务条款里说清楚。**
万一呢的角色是"安全传递加密数据"，不是"执行遗嘱"。信使拿到明文后怎么做，与平台无关。但如果信使用数据做违法的事……你需要在 ToS 中有限责条款。

**20. AGPL-3.0 对 SaaS 的影响。**
AGPL 要求修改后的代码即使只在服务端运行也要开源。如果你未来做 B2B 私有部署，企业客户会介意。考虑是否改为 AGPL-3.0 + 商业授权双许可。
